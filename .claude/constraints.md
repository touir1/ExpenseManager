## Key Constraints

Non-obvious things Claude cannot derive from reading files:

**nexus:** custom image at `infrastructure/configs/nexus/`; `provision.sh` runs in background at container startup via `docker-entrypoint.sh`; idempotent via `/nexus-data/.provisioned` flag; repo definitions in `repos.json` (direct REST API, no Groovy); DinD `command` array does NOT expand shell vars — `nexus:8082` hardcoded there; CI `script` lines use `$NEXUS_DOCKER_REGISTRY`; `NEXUS_USER`/`NEXUS_PASSWORD`/`NEXUS_NPM_TOKEN` are GitLab CI/CD secrets (not in any yml); Dockerfiles use `ARG REGISTRY=mcr.microsoft.com` — CI passes `--build-arg REGISTRY=$NEXUS_DOCKER_REGISTRY`.
**nginx auth:** subrequest `/internal/auth/check` → `users-service:9100/auth/check` before every request; only `/api/users/auth/` is public.
**nginx expenses routing:** `location /api/expenses` with `rewrite ^/api/expenses/?(.*)$ /$1 break` + `proxy_pass http://expenses-service:9200` — strips the `/api/expenses` prefix and forwards the remainder (e.g., `/api/expenses` → `/`, `/api/expenses/123` → `/123`, `/api/expenses/categories` → `/categories`); `ExpenseController` uses `[Route("")]` (service root) so expense CRUD lives at `/` internally; all other controllers (`CategoryController`, `CurrencyController`, `FamilyController`, `TagController`, `AdminRateController`, `AdminCategoryController`, `AdminCurrencyController`, `DashboardController`, `UserConfigController`, `ExpenseImportController`) keep their own route prefixes; `UserConfigController` uses `[Route("config")]` → `/api/expenses/config`; admin controllers use `[Route("admin/...")]` → `/api/expenses/admin/...`; `ExpenseImportController` uses `[Route("import")]` → `/api/expenses/import/...`.
**admin role guard (expenses):** `AppAdminAttribute` action filter calls `JwtCookieReader.GetIsAdmin(Request)` — decodes `isAdmin` JWT claim from `auth_token` cookie (or `Authorization: Bearer` header for Swagger); returns `403 { message: "FORBIDDEN" }` if false/missing. `JwtCookieReader` now has two public static methods: `GetUserId` and `GetIsAdmin`; both call `ExtractToken` internally (cookie → header fallback).
**admin role guard (users):** `AdminAuthorizeAttribute` action filter reads `isAdmin` JWT claim from the `auth_token` cookie using the same base64url decode as the expenses service; returns `403 { message: "FORBIDDEN" }` if false/missing.
**isAdmin JWT claim (users):** `JwtTokenService.GenerateTokenAsync` queries `IRoleRepository.GetUserRolesAsync(userId)` and adds claim `"isAdmin": "true"` when any role has `Code == "APP_ADMIN"`; absent otherwise. `GET /auth/session` response (`SessionResponse`) includes `IsAdmin: bool`. `POST /auth/login` response (`LoginResponse`) also includes `IsAdmin: bool` (same value, computed at login time) — frontend `AuthContext.login` merges this into user state so the admin navbar menu renders immediately without a page refresh.
**APP_ADMIN role:** `Role` table row with `Code="APP_ADMIN"`, `Name="App Administrator"` must be seeded manually (no EF Core data seed in migrations). Assign via `PUT /admin/users/{id}/roles` or direct DB insert into `URR_UserRoles`.
**cross-service DB:** expenses reads `ext.USR_Users` from users DB via `Repositories/External/UserRepository` (read-only, mapped in `ExpensesDbContext`); also `GetAdminUserIdsAsync()` — queries `ext.USR_Users WHERE USR_IsAdmin = true AND USR_IsDeleted = false` via EF Core; `USR_IsAdmin` column (migration `AddUserIsAdmin`) is kept in sync by the outbox: `user.created` sets it `false`; `user.updated` is published by `AdminUserService.SetUserRolesAsync` after every role change and carries the current `IsAdmin` value — expenses `UserEventConsumer` writes it to `ext.USR_Users` via `SaveOrUpdateUserAsync`.
**messaging:** RabbitMQ singleton `IRabbitMQService.GetConnection()`; expenses config env prefix: `EXPENSES_MANAGEMENT_EXPENSES_RABBITMQ_*`, users config prefix: `EXPENSES_MANAGEMENT_USERS_RABBITMQ_*`, notifications config prefix: `EXPENSES_MANAGEMENT_NOTIFICATIONS_RABBITMQ_*`; all services connect to vhost `expense_management`; `appsettings.json` takes priority over env vars for all options in both services — pattern: `GetValue("Key", envVar) ?? hardcoded`; `UserEventConsumer.ExecuteAsync` retries on `BrokerUnreachableException` every 5 s so host starts even if RabbitMQ not yet ready; users service publishes to `users.events` topic exchange (routing keys: `user.created` / `user.updated` / `user.deleted` for domain events; `user.email.verification.requested` / `user.password.reset.requested` / `user.password.changed` for email notification events — all via outbox); expenses service consumes queue `expenses.users.sync` (binding `user.#`) as `UserEventConsumer` BackgroundService → calls `IUserRepository.SaveOrUpdateUserAsync` or `DeleteUserAsync`; trigger: `RegistrationService.ValidateEmailAsync` calls `IUserRepository.ValidateEmailAsync` (saves user), then independently calls `IOutboxRepository.EnqueueAsync` to write the outbox event — no shared transaction; expenses `RabbitMQService` requires `DispatchConsumersAsync = true` for async consumer; expenses service also publishes to `expenses.events` topic exchange via outbox — routing keys `family.member.removed`, `family.invitation.requested`, `family.invitation.accepted`, `family.member.joined`, `family.expense.added`, `family.expense.deleted`, `expenses.import.completed`, `expenses.rate.conflict`; notifications service consumes `expenses.events` queue `notifications.expenses.sync` (bindings `family.#` AND `expenses.#`) as `FamilyEventConsumer` BackgroundService, AND consumes `users.events` queue `notifications.users.email` (binding `user.#`) as `UserNotificationEventConsumer` BackgroundService.
**outbox pattern (users):** `MSG_OutboxEvents` table stores unpublished events; `OutboxPublisherService` (BackgroundService) polls every 5 s, max 5 retries, calls `IUserEventPublisher.PublishRaw(eventType, payload, messageId)`; replay via `POST /messaging/replay?eventType=&from=&forceAll=`; stats via `GET /messaging/outbox/stats`.
**outbox pattern (expenses):** `OutboxEvents` table stores unpublished events; `FamilyOutboxPublisherService` (BackgroundService) polls every 5 s, max 5 retries, calls `IFamilyEventPublisher.PublishRaw(eventType, payload, messageId)`; written by: `FamilyService.RemoveMemberAsync` (member removed), `FamilyService.InviteAsync` (invitation requested — replaces former direct email), `FamilyService.AcceptInviteAsync` (invitation accepted + member joined events), `ExpenseService.AddAsync`/`DeleteAsync` (expense added/deleted for shared families), `CsvImportService.ConfirmImportAsync` (import completed), `CurrencyRateService.RunDailyUpdateAsync` (rate conflict); all enqueues are non-fatal try/catch so primary operations succeed even if outbox write fails; migration: `AddExpensesOutbox`.
**inbox pattern (expenses):** `InboxEvents` table deduplicates consumed messages; `UserEventConsumer` extracts `MessageId` from `ea.BasicProperties.MessageId` (fallback: new GUID); checks `IInboxRepository.ExistsAsync` before processing — duplicate → ack and skip; on success writes `InboxEvent { Status="Processed" }` then acks; on failure nacks without inbox write.
**inbox pattern (notifications):** `InboxEvents` table deduplicates all consumed messages; `FamilyEventConsumer` and `UserNotificationEventConsumer` follow the same deduplication pattern as expenses `UserEventConsumer` — extract `MessageId` from `ea.BasicProperties.MessageId` (fallback: new GUID), check `IInboxRepository.ExistsAsync` before processing; `OnMessageReceivedAsync` and `Ack`/`Nack` are `protected` + virtual so `TestableConsumer` can override without a real channel.
**notifications service (Phase 13+14):** dedicated microservice at port 9300; **sole SMTP email dispatcher for the entire platform** — users and expenses services no longer send email directly; `NotificationsDbContext` holds `Notifications` + `InboxEvents` tables; `UseIdentityAlwaysColumn()` and partial index guarded behind `Database.IsNpgsql()` for SQLite test compatibility; `NotificationHub` (SignalR) at `/ws/notifications` — authenticates via `auth_token` cookie (`JwtCookieReader.GetUserId`), groups connections by `userId.ToString()`; handler dispatch pattern: persist DB (non-fatal try/catch) → hub push (non-fatal try/catch) → email (non-fatal try/catch); email-only handlers (email verification, password reset, password changed, family invitation) skip DB persist and hub push; fan-out handlers (member joined, expense added/deleted, rate conflict) loop per recipient with independent try/catch per step; two consumers: `FamilyEventConsumer` (queue `notifications.expenses.sync`, bindings `family.#` + `expenses.#`, handles 8 event types) and `UserNotificationEventConsumer` (queue `notifications.users.email`, binding `user.#`, handles 3 email-only event types); `FamilyEventConsumer.OnMessageReceivedAsync` first deserializes to `BaseEventEnvelope` (just `EventType`) then switches and deserializes to the specific DTO per case in `HandleMessageAsync`; event message payload stored as JSON in `Notification.Payload`; env prefix `EXPENSES_MANAGEMENT_NOTIFICATIONS_DATABASE_*` for Postgres, `EXPENSES_MANAGEMENT_NOTIFICATIONS_RABBITMQ_*` for RabbitMQ, `EXPENSES_MANAGEMENT_NOTIFICATIONS_EMAILAUTH_*` for SMTP; nginx routes `/api/notifications/ws/` (WebSocket, 3600s timeout) and `/api/notifications` (REST) to `notifications-service:9300`; frontend connects via `@microsoft/signalr` `HubConnectionBuilder` with `withCredentials: true`; 6 email templates in `Assets/EmailTemplates/` (all as EmbeddedResource): `FAMILY_MEMBER_REMOVED_TEMPLATE.html`, `FAMILY_INVITATION_TEMPLATE.html`, `FAMILY_INVITATION_ACCEPTED_TEMPLATE.html`, `EMAIL_VERIFICATION_TEMPLATE.html`, `PASSWORD_RESET_TEMPLATE.html`, `PASSWORD_CHANGED_TEMPLATE.html`.
**backend tests (notifications):** use `TestNotificationsDbContextWrapper` (SQLite in-memory, `EnsureCreated`) — same pattern as expenses but EnsureCreated not Migrate (no migrations in test path); `TestableConsumer` subclass exposes `InvokeOnMessageReceived` and overrides `Ack`/`Nack` to capture call state without a real channel.
**DB migrations:** auto-run at startup via `MigrateAsync()`.
**OutboxEvent.Id (users) — provider-specific config:** `OutboxEvent.Id` is `long`; `UsersAppDbContext.OnModelCreating` applies `UseIdentityAlwaysColumn()` only when `Database.IsNpgsql()`, and `ValueGeneratedOnAdd()` otherwise — this lets `EnsureCreated()` on SQLite emit `INTEGER PRIMARY KEY` (auto-increment) for tests; Npgsql's `IdentityAlwaysColumn` annotation forces `BIGINT` even on SQLite, breaking test auto-increment; `TestDbContextEnsureCreated` uses this path (never call `Migrate()` for `OutboxRepositoryTests`).
**user soft-delete:** `User` has `IsDeleted` (bool, default false) + `DeletedAt` (DateTime?); `UserRepository.DeleteUserAsync` sets flags — never calls `.Remove()`; all five repository query methods filter `!u.IsDeleted`; partial unique index `ux_usr_email_active` on `USR_Email WHERE USR_IsDeleted = FALSE` enforces uniqueness only among active users — same email can be reused after deletion; `IsDisabled` is separate (account suspension, not deletion).
**email verification expiry:** `User.EmailValidationHashExpiresAt` (`DateTime?`, column `USR_EmailValidationHashExpiresAt`) — set to `UtcNow + 24 h` on registration and resend; `ValidateEmailAsync` returns `null` (→ error page) when value is non-null and past `UtcNow`; null = no expiry (legacy rows); `UpdateEmailValidationHashAsync` atomically sets hash + expiresAt — old link immediately dead; `ResendVerificationEmailAsync` always returns `200 OK` regardless of account existence (no leak); re-registering an unverified email silently triggers resend — no error exposed to caller.
**rate limiting (users service):** built-in .NET 8 `Microsoft.AspNetCore.RateLimiting`; fixed-window per client IP; policies: `login` (10/1 min), `register` (5/10 min), `resend_verification` (3/10 min), `validate_email` (10/5 min), `request_password_reset`/`change_password_reset`/`create_password` (5/10 min), `change_password` (10/5 min), `refresh` (20/1 min), `messaging_replay` (5/1 min); no rate limiting on `GET /auth/check`, `GET /auth/session`, `POST /auth/logout`, `GET /messaging/outbox/stats`.
**rate limiting (expenses service):** `expenses_global` sliding window 100 req/60 s, 6 segments, per client IP; applied at class level on all three controllers (`CategoryController`, `CurrencyController`, `ExpenseController`) via `[EnableRateLimiting("expenses_global")]`.
**family system (expenses):** `FamilyService.CreateDefaultAsync(userId)` is idempotent — checks `HasDefaultFamilyAsync` before creating; called by `UserEventConsumer` on `user.created` after `SaveOrUpdateUserAsync`; `FamilyInvitation` table stores invite tokens (GUID); invite expiry is configurable via `FamilyOptions.InviteExpiryInDays` (env: `EXPENSES_MANAGEMENT_EXPENSES_FAMILY_INVITE_EXPIRY_IN_DAYS`, default 7); `InviteAsync` enqueues outbox event (`family.invitation.requested`) after persisting the invitation — no direct email call; invite URL built from `FamilyOptions.InviteBaseUrl` (env: `EXPENSES_MANAGEMENT_EXPENSES_FAMILY_INVITE_BASE_URL`, default `https://localhost/families/accept-invite`) + `?token={token}`; `AcceptInviteAsync` validates token not expired/already accepted and that acceptor's email matches `InviteeEmail`; `ArchiveAsync` = soft delete (`IsDeleted=true`) on non-default family only; `RemoveMemberAsync` also calls `RemoveMemberAttributionsAsync(familyId, ownerId)` to purge that member's expense attributions for the family; expense `FamilyIds int[]?` in Add/Update requests — null → auto-attribute to user's default family; empty → no attribution; provided → validate membership per familyId; `FamilyForbiddenException` (→ 403 in FamilyController and ExpenseController), `FamilyNotFoundException` (→ 404), `FamilyConflictException` (→ 409), `FamilyInvitationException` (→ 400); role name→ID resolution via `ILookupCacheService.GetIdAsync<FamilyRole>` — never hardcoded IDs; role name normalised to title case before lookup. **Name uniqueness:** `IFamilyRepository.ExistsWithNameForUserAsync(name, userId, excludeId?)` checks case-insensitively across the user's non-deleted family memberships; `CreateAsync` and `RenameAsync` throw `FamilyConflictException("FAMILY_NAME_ALREADY_EXISTS")` → 409 on collision; frontend `FamiliesPage.tsx` handles 409 by calling `setError('name', …)` on the modal form.
**family system (expenses service):** `IUserRepository.GetUserByEmailAsync` looks up user by email (for invite flow); `GetUserByIdAsync` now filters `!u.IsDeleted`; `IFamilyRepository` injected into `ExpenseService` for attribution writes; `IUserRepository.GetAdminUserIdsAsync()` queries `ext.USR_Users.USR_IsAdmin` column (used by `CurrencyRateService` to populate `RateConflictEventMessage.AdminUserIds`).
**tag system (expenses):** `Tags` table is global — no per-user ownership; unique by `Name` (case-sensitive, `HasIndex(e => e.Name).IsUnique()`); `UserTags` junction `(UserId, TagId)` composite PK tracks adoption — a UserTag row means "this user has adopted this tag"; adoption created via `POST /tags` OR auto-adopt when attaching tag to an expense; `ITagRepository.GetFamilyAsync` returns tags adopted by co-members of shared non-deleted families, excluding tags already in own; `IsVisibleAsync` = own UserTag OR co-member UserTag; `FamilyForbiddenException` (→ 403) used for invisible tag violations; `ExpenseService.WriteExpenseTagsAsync` validates visibility then auto-adopts then inserts `ExpenseTag` rows; `ExpenseService.MapToDto` takes optional `IEnumerable<TagDto>? explicitTags` — used for add/update (nav not loaded), falls back to `e.ExpenseTags` for read paths; Moq CS0854 workaround: audit interface methods keep optional `string tags = ""` defaults but test Verify calls explicitly pass `It.IsAny<string>()` to avoid expression-tree compilation failure.
**category name uniqueness (expenses):** `ICategoryRepository.ExistsWithNameAsync(name, parentCategoryId, excludeId?)` checks case-insensitively for duplicate name at the same level (top-level: `parentCategoryId=null`; subcategory: `parentCategoryId=parentId`); `AdminCategoryService` calls this before `AddAsync`/`UpdateAsync` in all four create/update methods — throws `InvalidOperationException("CATEGORY_NAME_DUPLICATE")` → controller maps to 400; scope is per-level (subcategory names only need to be unique within their parent, not globally).
**expenses soft-delete (`Category`, `Family`, `Expense`):** all three use `IsDeleted` (bool, default false) + `DeletedAt` (DateTime?); `CategoryRepository.GetAllActiveAsync` filters `!c.IsDeleted` on top-level; `CategoryService.GetAllAsync` filters subcategories by `!s.IsDeleted`; `ExpenseRepository` filters `!e.IsDeleted` in both `GetByIdAsync` and `GetPagedAsync`; `SoftDeleteAsync` sets flags — never calls `.Remove()`; no family repository delete path yet; expenses raw SQL migrations use `CURRENT_TIMESTAMP` (not `NOW()`) to remain compatible with SQLite test runner.
**backend tests:** use `TestExpensesDbContextWrapper` (in-memory DB, expenses); `TestDbContextWrapper` (SQLite + Migrate, users — for entities with int/short PKs); `TestDbContextEnsureCreated` (SQLite + EnsureCreated, users — for `OutboxEvent` which has `long` PK); `TestNotificationsDbContextWrapper` (SQLite + EnsureCreated, notifications); never mock DbContext.
**DbContext concurrency:** EF Core's scoped `DbContext` does NOT support concurrent operations — `Task.WhenAll` with two repository methods sharing the same context throws `InvalidOperationException: A second operation was started on this context instance before a previous operation completed`. Always await repository calls sequentially within a single service method.
**frontend auth:** `credentials: 'include'` on all requests; session restored via `GET /auth/session` (returns `{ email, firstName, lastName }`); if session check fails, `AuthContext` explicitly calls `POST /auth/refresh` then retries session check; all HTTP in `features/auth/services/authApi.service.ts`, state in `features/auth/AuthContext.tsx`; no `localStorage`/`sessionStorage` for auth data.
**logout navigation:** `NavBar.handleLogout` sets `loggingOutRef.current = true` (a `useRef`, not state) then calls `logout()`; a `useEffect` watching `[isAuthenticated, navigate]` calls `Promise.resolve().then(() => navigate('/'))` when `loggingOutRef.current && !isAuthenticated`. The microtask deferral is critical: with `v7_startTransition: true`, React Router effects from `ProtectedRoute`'s `<Navigate to="/login" replace />` fire in the same commit as NavBar's effect (NavBar is earlier in the tree, so its effect runs first). Without the microtask, NavBar calls `navigate('/')` (push:/) then ProtectedRoute's Navigate fires `replaceState:/login`, overwriting it. With `Promise.resolve()`, the navigate fires after all same-commit effects, so it overrides the replaceState and the user lands on `/`. Do NOT use `useState` for the loggingOut flag — an urgent state update interrupts React Router's startTransition and causes similar redirect races. NavBar logout tests call `rerender` with `{ isAuthenticated: false }` inside `await act(async () => {...})` to flush both the effect and the microtask.
**refresh token:** backend stores opaque tokens in `RTK_RefreshTokens`; `api.service.ts` transparently retries any 401 non-`skipUnauthorized` request via `POST /auth/refresh` (deduplicates concurrent refreshes); `skipUnauthorized: true` disables both the retry and the `unauthorizedHandler`; `silent: true` additionally suppresses the `errorHandler` toast — use for startup probes (`sessionCheck`, `refreshRequest`) that are expected to fail for unauthenticated visitors; when `unauthorizedHandler` is null (e.g., after `logout()` calls `onUnauthorized(null)`), any 401 after refresh failure returns `{ ok: false, status: 401 }` silently with no redirect — racing in-flight requests after sign-out do not hard-navigate to `/login`; `redirectToLogin()` is `unauthorizedHandler?.()` — no fallback.
**frontend forms:** all auth forms use `useForm<T>({ resolver: zodResolver(...) })` from React Hook Form; Zod schemas and inferred types live in `src/features/auth/auth.schemas.ts`; per-field errors use `aria-describedby` with field-scoped IDs (e.g. `email-error`); `PasswordInput` uses `forwardRef` so RHF's ref callback reaches the underlying `<input>`.
**frontend CSS:** shared primitives in `@layer components` in `index.css` — extend before adding new classes; brand `brand-600`; cards `bg-white shadow-card border border-slate-200 rounded-2xl`.
**backend validation:** all controller Request DTO validation uses FluentValidation (`AddFluentValidationAutoValidation()` + `AddValidatorsFromAssemblyContaining<Program>()`); `InvalidModelStateResponseFactory` returns `401 UnauthorizedObjectResult(new ErrorResponse { Message = firstError })` to preserve the existing wire format; validators live in `Validators/` and use `CascadeMode.Stop` per-field; `repeatPassword`/`ConfirmPassword` is NOT in backend DTOs — it is a frontend-only UX concern.
**expenses lookup tables:** all enum-like domain values (operation source, family role, rate source, etc.) are DB lookup tables in `Models/Lookups/`, not C# enums; FK columns are `int` (e.g. `CreatedFromId`, `RoleId`); use `ILookupCacheService.GetIdAsync<T>(name)` / `GetNameAsync<T>(id)` to resolve — backed by `IMemoryCache` with `NeverRemove` priority; seed IDs: OperationSource 1=SingleWeb/2=SingleMobile/3=BulkWeb, ModifiedSource 1=Web/2=Mobile, FamilyRole 1=Head/2=Member, RateSource 1=Auto/2=Manual, ConflictStatus 1=Pending/2=Resolved, ConflictResolution 1=AcceptAuto/2=KeepManual/3=Custom, AuditOperation 1=Add/2=Update/3=Delete, SnapshotType 1=Before/2=After.
**expenses controller structure:** mirrors users service — `Controllers/DTO/` for reusable data objects (e.g. `CategoryDto`, `CurrencyDto`), `Controllers/Requests/` for request bodies, `Controllers/Responses/` for envelope responses and `ErrorResponse`; services return DTO types, not model types; error string constants split into two internal static classes: `Controllers/ControllerErrors.cs` (HTTP-layer codes, same namespace as controllers — no using needed) and `Services/ServiceErrors.cs` (domain exception codes thrown by services — `FAMILY_*`, `TAG_NOT_VISIBLE`, `USER_NOT_FOUND`, invitation codes); both are expenses-only; users service has no equivalent (no domain exceptions).
**201 Created responses (expenses controllers):** use `CreatedAtRoute(routeName, routeValues, value)` — NOT `CreatedAtAction`; `MvcOptions.SuppressAsyncSuffixInActionNames` defaults to `true`, so `nameof(GetByIdAsync)` → lookup for `"GetByIdAsync"` fails (descriptor is registered as `"GetById"`) → `InvalidOperationException` at response formatting; named routes (`Name = "GetFamilyById"` / `Name = "GetExpenseById"` on the `[HttpGet]` attribute) are unaffected by this stripping.
**expense userId extraction:** `JwtCookieReader.GetUserId(Request)` reads `auth_token` cookie, base64url-decodes the JWT payload, extracts `sub` claim as int — no signature validation (nginx already validated); no `System.IdentityModel.Tokens.Jwt` package in expenses project; if cookie missing or decode fails returns null → controller returns 401.
**ExpenseDto shape:** uses nested objects — `Currency: CurrencyDto?`, `Category: SubcategoryDto?`, `Subcategory: SubcategoryDto?`, `Tags: IEnumerable<TagDto>`, `Families: IEnumerable<FamilyNameDto>` (non-default attributions only) — not flat `CurrencyId`/`CategoryName` fields; `SubcategoryDto` (`{ id, name, description }`) reused for both category and subcategory slots because it has no children collection; `FamilyNameDto` (`{ id, name }`) used for the families list; `ExpenseService.MapToDto` accepts `explicitTags` and `explicitFamilies` params — both used for Add/Update paths because EF Core relationship fixup populates nav collections after insert but without sub-nav props loaded (e.g. `a.Family` is null on a freshly inserted `ExpenseFamilyAttribution`), which causes NullReferenceException during JSON serialization; read paths (`GetByIdAsync`, `GetPagedAsync`) use loaded nav props with null guard; `ExpenseFamilyAttributions` nav prop must be included in `GetByIdAsync` and `GetPagedAsync` queries to populate `Families`; `WriteAttributionsAsync` returns `(IReadOnlyList<int> AllIds, IReadOnlyList<FamilyNameDto> NonDefaultFamilies)` — explicit-family path uses `GetFamiliesByUserAsync` (single query, filters `!IsDeleted`) for membership validation and name resolution instead of N `IsMemberAsync` calls.
**expense audit:** `ExpenseAuditService` (injected into `ExpenseService`, never exposed via controller) writes `ExpenseAuditLog` then `ExpenseAuditSnapshot(s)` in two `SaveChangesAsync` calls; add→1 after snapshot, update→before+after, delete→1 before; snapshot stores comma-separated tag/family IDs (empty for Phase 3); `PerformedFromId` uses `OperationSource` seed IDs (1=SingleWeb hardcoded for web controller).
**backend validation (expenses validators):** use `ClassLevelCascadeMode = CascadeMode.Stop` (not the deprecated `CascadeMode = CascadeMode.Stop`); `SubcategoryId` requires `CategoryId` — validated via `.Must((req, subId) => subId == null || req.CategoryId != null)`; shared rules live in `ExpenseRequestValidatorBase<T> where T : IExpenseRequest` — both concrete validators are empty subclasses; value-type properties on request DTOs use `required` modifier to prevent under-posting.
**expenses data provider:** `ExpensesDataProvider` gates all API fetches on `isAuthenticated` from `useAuth()`; fetching without auth causes 401 → transparent refresh → `location.assign('/login')` → infinite reload loop; `isLoading` starts `false` and only becomes `true` during an active fetch; exposes `categories`, `currencies`, `tags` (own + family, deduped by id via `getTags()`) and `refresh`; always destructure `tags` with a `= []` default in consumers to guard against incomplete mocks in tests.
**ExpensesPage family filter:** `ExpensesPage` reads `activeFamilyId` from `useFamilies()` (FamilyContext) and merges it into `effectiveFilter: ExpenseFilter` — `{ ...filter, familyId: activeFamilyId ?? undefined }` — passed to `getExpenses` and included in the `queryKey`. A `useEffect` watching `activeFamilyId` resets `filter.page` to 1 on family change. Backend: `ExpenseFilterDto.FamilyId?: int?` — `ExpenseRepository.GetPagedAsync` adds `.Where(e => e.ExpenseFamilyAttributions.Any(a => a.FamilyId == filter.FamilyId && !a.Family.IsDeleted))` when set. Personal view (`null`) shows all user expenses; family view shows only expenses attributed to that family.
**notifications frontend:** `NotificationContext.tsx` uses dynamic `import('@microsoft/signalr')` inside a `useEffect` to build the `HubConnection`; mock with a real `class MockHubConnectionBuilder` (not `vi.fn().mockReturnValue({...})`) — `vi.fn().mockReturnValue()` sets an arrow function `() => value` as implementation; arrow functions cannot be called with `new`, causing `TypeError: () => value is not a constructor`. `getNotificationText(n, t)` in `NotificationBell.tsx` centralises all 7 type→label mappings; extend it when adding new notification types. In tests: `screen.getAllByText(matcher).length > 0` rather than `screen.getByText(matcher)` when the text appears in both a `<p>` and its parent `<button>` (both have matching `textContent`).
**frontend tests:** `toHaveClass()` not `toHaveStyle()` — Tailwind doesn't produce inline styles in jsdom. Never mix `fireEvent` and `userEvent.setup()` in the same test — use one or the other exclusively or the internal DOM model gets out of sync. `toLocaleString()` output is locale-dependent — GitLab CI runner uses C/POSIX locale (comma thousands separator, e.g. `2,430.50`) while dev machines may use French locale (space separator, `2 430,50`). When asserting on `toLocaleString()` output, always strip both whitespace AND commas: `text.replace(/[\s,]/g, '').includes('2430')` — `\s` covers all Unicode whitespace including U+202F (narrow no-break space used by FR locale), comma covers en-US/CI. `<input type="date">` does NOT fire React's `onChange` via `userEvent.type()` in jsdom — use `fireEvent.change(input, { target: { value: '...' } })` instead. `FormCombobox` (used in `ExpenseForm` for currency/category/subcategory): to interact in tests use `fireEvent.focus(input)` to open the dropdown, then `fireEvent.mouseDown(getByRole('option', { name: '...' }))` to select — do NOT use `user.selectOptions()` (that's for native `<select>`).
**Zod v4 (project uses v4.x):** `required_error` and `invalid_type_error` params removed — use `error: message` instead (e.g. `z.number({ error: t('validation.required') })`). `z.preprocess` loses type inference in v4 — use `.catch(defaultValue)` instead to coerce bad values (e.g. `z.number().optional().catch(undefined)` converts NaN from disabled RHF selects to `undefined`). `z.preprocess` is deprecated; prefer `z.pipe` or `.catch()`.
**RHF + disabled selects:** `register('field', { valueAsNumber: true })` on a disabled `<select>` reads `""` on submit → `parseFloat("") = NaN`. Zod v4 rejects NaN from `z.number()`. Fix: `.catch(undefined)` on the Zod field so NaN silently becomes `undefined` before validation.
